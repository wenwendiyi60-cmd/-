# 六爻预测口令系统 · 版本管理仓库

> 本仓库是一套**严格可审计、版本可追溯**的六爻预测 AI 口令工程系统。  
> 这不是普通的编程项目，而是一套用于管理六爻断卦 AI 口令的规则与版本控制体系。

---

## 核心立场

| 原则 | 说明 |
|------|------|
| 未知即未知 | 信息缺失时，拒绝给出结论，不以模糊语言掩盖不确定性 |
| 不造动爻 | 动爻必须由用户明确提供，不得推测 |
| 不造旬空 | 旬空依据起卦日干支计算，不得默认或凭空列出 |
| 不造变卦 | 无动爻则无变卦，不得自行生成 |
| 阶段不得跳跃 | 第一阶段盘面未确认前，不得进入分析 |
| 过程证据不替代结论证据 | 分析过程的推理不等于最终断语的依据 |

## 口令版本管理规范

1. **升级前**：将当前 `active/` 中的文件复制到 `archive/`，注明版本号与日期
2. **修改时**：在 `templates/prompt-template.md` 基础上编写新版本
3. **发布后**：在 `changelog.md` 顶部追加记录
4. **测试**：每次发布前用 `tests/` 中的回归测试验证口令行为

---

## 防幻觉快速检查

提交任何口令修改前，确认：

- [ ] 口令是否要求第一阶段盘面确认后才能分析？
- [ ] 口令是否明确禁止在无动爻时生成变卦？
- [ ] 口令是否在用户未提供日期时拒绝列出旬空？
- [ ] 口令是否区分"过程证据"与"结论证据"？
- [ ] 口令是否允许输出"信息不足，无法断定"？
